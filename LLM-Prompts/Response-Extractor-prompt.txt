Section meanings:
- system_summary: general summary of the LLM-based system.
- architecture: concrete architecture objects that can later become DFD nodes, edges, or trust boundaries.
- security_controls: implemented safeguards only.
- risk_signals: misuse cases, operational weaknesses, and candidate OWASP LLM risks.

Classification rules:
- Actors are human users, external systems, or services that initiate interactions.
- The system purpose itself must not be classified as an actor.
- Components are application, backend, API, model, orchestration, RAG, or tool-execution units.
- Data stores are persistent storage systems, logs, vector stores, databases, file stores, knowledge bases, or conversation history stores.
- External services are third-party APIs, external model providers, SaaS integrations, email providers, notification services, or external web services.
- Trust boundaries must be explicit first-class objects when implied by the answers.
- Data flows must include source, destination, exposure, and trust_zone.
- Security controls must include only implemented safeguards.
- Threats, attack scenarios, misuse scenarios, and weaknesses must go under risk_signals.
- Do not put prompt injection, sensitive data extraction, unauthorized action triggering, misinformation, unsafe recommendations, denial of service, excessive usage, data leakage, prompt leakage, jailbreak, poisoning, or abuse under security_controls.
- Add confidence where useful: high, medium, low.
- If an item is inferred rather than directly stated, include "inferred": true.

Keep items compact. Prefer high-confidence extraction over speculation.
